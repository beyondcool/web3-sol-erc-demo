## ERC-8183 是什么？三句话讲清楚

| 问题 | 答案 |
|------|------|
| **它解决什么问题？** | AI Agent 之间要互相**雇佣和付钱**，但没有平台担保，怎么**安全交易**？— 我先付钱怕跑路，我先干活怕白干。 |
| **怎么做？** | 引入 **Job 托管+验收** 机制：Client 把钱锁在合约里（托管），Provider 干活提交，Evaluator（可以是另一个 Agent 或合约）验收，合约自动放款或退款。全程无需信任任何一方。 |
| **和别的协议什么关系？** | **x402** 解决怎么付（小额支付），**ERC-8004** 解决 Agent 的身份和声誉（谁可信），**ERC-8183** 解决怎么安全交易（付钱+验收）。三者组成 AI Agent 经济的完整栈。 |

## 核心概念

### 一个 Job，三个角色

```
┌──────────────────────────────────────────────────────────────────────────┐
│                           ERC-8183 Job                                    │
│                                                                           │
│  ┌──────────┐    发布任务+出钱       ┌────────────┐                       │
│  │  Client  │ ──────────────────▶  │            │                       │
│  │ (Alice)  │   setBudget + fund   │  Contract  │  ┌─────────────────┐  │
│  └────┬─────┘                      │  (Escrow)  │  │   Evaluator     │  │
│       │                            │            │  │   (Eva — 仲裁)   │  │
│       │ refund                     │            │  │                  │  │
│       ◀────────────────────────────│            │  │  complete /      │  │
│       │                            │            │◀─│  reject          │  │
│       │                            │            │  └─────────────────┘  │
│  ┌────┴─────┐                      │            │                       │
│  │ Provider │                      │ 提交交付物   │                       │
│  │ (Bob)    │ ──────────────────▶  │            │                       │
│  └──────────┘  submit + 拿钱        │   ◀─────── │                       │
│         ◀──────────────────────────┘   payment  │                       │
└──────────────────────────────────────────────────────────────────────────┘
```

| 角色 | 做什么 | 关键能力 |
|------|--------|---------|
| **Client** | 发布任务 + 出钱 | setBudget / fund / reject（仅 Open） |
| **Provider** | 干活 + 交活 | setBudget（议价） / submit |
| **Evaluator** | 验收 | complete / reject（Funded/Submitted） |

> ⚡ **Evaluator 可以是任何人/东西**：AI Agent、ZK 验证合约、多签钱包、DAO——协议只关心**谁调用了 complete 或 reject**。

### 六种状态的状态机

```
                    ┌──────────┐
                    │   Open   │  ← 创建 Job
                    └────┬─────┘
                   ╱     │     ╲
                 ╱       │       ╲
           fund ╱        │reject  ╲
               ╱         │         ╲
         ┌────────┐   Rejected     ╱
         │ Funded │  （终态）     ╱
         └───┬────┘              ╱
             │                  ╱
         submit               ╱
             │               ╱
         ┌──────────┐       ╱
         │Submitted │ ←────╱
         └──┬───────┘
        ╱    │     ╲
       ╱     │      ╲
complete ╱   │reject ╲ 过期
     ╱       │        ╲
┌────────┐ ┌────────┐ ┌────────┐
│Completed│ │Rejected│ │Expired │
│  （终态） │ │  （终态） │ │  （终态） │
└────────┘ └────────┘ └────────┘
  资金给       资金退还      资金退还
  Provider     Client       Client
```

### 关键机制

**🔒 托管（Escrow）**
Client 调用 `fund` 时，资金锁定在合约中，谁也不能单方面拿走。要么 Evaluator 验收通过→给 Provider，要么拒绝/过期→退给 Client。

**⚖️ Evaluator 仲裁**
协议不定义"什么是好的工作质量"，这交给 Evaluator。Evaluator 的 `complete` 或 `reject` 决定资金去向。

**🛟 过期逃生（claimRefund）**
如果 Evaluator 掉线或迟迟不验收，任何人在 `expiredAt` 之后都可以调用 `claimRefund` 触发退款——资金永不卡死。此函数不可被 Hook 拦截。

**🤝 价格协商（setBudget）**
Client 和 Provider 都可以调用 `setBudget`。一个出价，一个还价，最后 Client 的 `fund` 决定是否接受。

### Extensions（本教学示例未实现）

真实 ERC-8183 还有两个扩展机制，教学版省略以保持焦点：

- **Fee（平台费）**：每次 `complete` 可以扣除一定比例的平台费给 treasury
- **Hook（钩子系统）**：在 6 个核心函数前后插入自定义逻辑（KYC、竞价、保险等）。`claimRefund` 不可被 Hook 拦截——这是设计上的安全底线。

## 教学示例 vs 真实 ERC-8183

| 方面 | 教学版 | 真实 ERC-8183 |
|------|--------|---------------|
| **平台费** | 未实现（简化） | complete 时可扣 fee 给 treasury |
| **Hook 系统** | 未实现（仅注释提及） | IACPHook 接口，6 个函数的 before/after 回调 |
| **Payment Token** | 全局统一（构造时指定） | 可为每个 Job 指定不同代币 |
| **Job 描述** | 纯文本 string | 支持更丰富的数据结构 |
| **setProvider** | 仅有基础的设置 | 完整的前后端 Hook 支持 |

## 与 ERC-8004 的关系

ERC-8183 与 ERC-8004 互补构成"**发现→交易→声誉**"的闭环：

```
  发现         交易         声誉
┌────────┐  ┌────────┐  ┌────────┐
│8004    │─▶│8183    │─▶│8004    │
│身份注册 │  │任务托管 │  │打分评价 │
└────────┘  └────────┘  └────────┘
```

每次成功完成一个 Job，链上都留下不可篡改的记录（交付物哈希、Eva 的 attestation、结果），这些记录可以喂给 ERC-8004 的声誉系统，让做得好的 Agent 更容易被找到和信任。
