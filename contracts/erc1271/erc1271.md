## 创建的文件

### 1. contracts/erc1271/ERC1271Wallet.sol

一个简洁的合约钱包，实现了 ERC-1271 标准。贯穿全文的中文注释讲解了：

- **为什么需要 ERC-1271** — EOA 有私钥可以签名，合约是代码没有私钥，需要标准化接口
- **协议核心** — 只有一个函数 `isValidSignature(bytes32 hash, bytes calldata signature) → bytes4`，返回 `0x1626ba7e` 表示有效
- **能做什么** — 合约钱包、统一验证（EOA + 合约）、链下授权

合约逻辑极简：有一个 `owner`，`isValidSignature` 用 `ecrecover` 恢复签名者，判断是否匹配 owner。

### 2. contracts/erc1271/ERC1271Wallet.t.sol

四个测试，逐层递进：

| 测试                                     | 教学目的                                         |
| -------------------------------------- | -------------------------------------------- |
| `test_ValidOwnerSignature`             | owner 签名 → 合约接受（返回魔法值）                       |
| `test_RejectNonOwnerSignature`         | 非 owner 签名 → 合约拒绝                            |
| `test_ContractDefinesItsOwnValidation` | 合约**自定义**验证逻辑——这是 ERC-1271 和 ecrecover 的本质区别 |
| `test_UnifiedVerification`             | 外部系统统一验证，不关心合约内部逻辑——ERC-1271 的真正价值           |

**核心教学点：**

- `ecrecover` 只验 EOA，ERC-1271 让**合约也能验签名**
- 合约可以自定义验证逻辑（单签/多签/社交恢复），**接口不变**
- 外部系统无需区分 EOA 还是合约，统一调用 `isValidSignature` 即可


简单来说，**ERC-1271 并不是让智能合约“自己签名”，而是定义了一套标准，让智能合约能够“验证一份签名是否有效”**。签名动作仍然由持有私钥的实体在链下完成，合约只是根据自身的逻辑去判断这个签名能否代表“合约账户”的意愿。

---

### 1. EOA 的签名与验证（回顾）

- **签名**：用私钥对消息哈希进行 ECDSA 签名，产生 `(r, s, v)`。
- **验证**：任何人都可以用消息哈希、签名和公钥（或地址），通过 `ecrecover` 还原出签名者地址，然后比对是否匹配。

这是完全基于密码学的，不需要任何链上代码逻辑。

---

### 2. 智能合约的困境

智能合约没有私钥，无法生成 ECDSA 签名。但如果想实现“合约账户”的功能（比如多签钱包、社交恢复钱包），就需要有一种方式，让外部合约能够问这个合约：  
**“这份签名，你认不认？”**

ERC-1271 就是这一问一答的标准接口。

---

### 3. ERC-1271 的核心：`isValidSignature`

ERC-1271 要求合约实现一个函数：

```solidity
function isValidSignature(bytes32 _hash, bytes memory _signature) 
    public view returns (bytes4 magicValue);
```

- **`_hash`**：需要验证的消息哈希。
- **`_signature`**：任意格式的“签名”数据，由合约自己定义如何解析。
- **返回值**：如果签名有效，**必须**返回 `0x1626ba7e`（即 `bytes4(keccak256("isValidSignature(bytes32,bytes)")` 的前 4 字节），否则返回其他值（如 `0xffffffff`）表示无效。

**魔法值 `0x1626ba7e` 的作用**：防止因函数签名碰撞或默认返回值被误认为验证通过。

---

### 4. 签名和验证的完整流程

整个过程分为链下生成签名和链上验证两步。

#### 链下：生成“签名”

这里的“签名”已经不局限于 ECDSA 的 `(r,s,v)` 了，而是合约自己定义的一串字节。

举例：

- **单签名合约**（如简单的智能钱包，由 1 个 EOA 控制）：  
  控制者用自己的私钥对 `_hash` 做 ECDSA 签名，得到 `(r,s,v)`，然后把它们拼接起来作为 `_signature`。

- **多签合约**（如 Gnosis Safe）：  
  多个控制者分别用各自私钥签名同一个 `_hash`，把这些签名按顺序打包成 `_signature`。  
  格式可能是：`[签名1][签名2]...` 或带有长度的变长编码。

- **更复杂的逻辑**：甚至可以完全不依赖 ECDSA，比如基于时间锁、角色权限等逻辑，`_signature` 里只是一些标识，合约内部自行判断。

#### 链上：合约验证

调用者（比如一个 DApp）拿着 `_hash` 和 `_signature` 去调用合约的 `isValidSignature`。

合约内部会：

1. 解析 `_signature`，提取出其中的签名数据或标识；
2. 按照自己的规则验证：
   - 若是单签，用 `ecrecover` 还原地址，检查是否等于预设的 `owner`；
   - 若是多签，逐个 `ecrecover` 还原出签名者地址，检查是否达到门限，且无重复；
3. 如果规则全部满足，返回 `0x1626ba7e`；否则返回其他值。

这样，调用者无需知道合约内部逻辑，只要看到返回魔法值，就认为“这个合约账户认同了这份消息”。

---

### 5. 实际应用场景

很多用到“账户抽象”的场景都依赖 ERC-1271：

- **ERC-4337 账户抽象**：用户操作 (UserOp) 的验证阶段，智能合约钱包需要实现 `validateUserOp`，其底层签名验证也常与 ERC-1271 兼容。
- **ERC-2612 无 gas 授权（Permit）**：代币合约支持 permit 时，如果 `owner` 是合约地址，它会通过调用 `owner` 的 `isValidSignature` 来验证授权签名，而不是直接用 `ecrecover`。这使得多签、智能钱包也能使用 permit。
- **登录验证**：DApp 可以让用户用合约钱包“签名”登录，后台通过 `isValidSignature` 验证。

### 总结对比

|         | EOA                     | 支持 ERC-1271 的合约                     |
| ------- | ----------------------- | ----------------------------------- |
| 私钥      | 有，直接签名                  | 无，由控制它的 EOA 在链下签名                   |
| 签名生成    | 私钥签名消息哈希，输出 `(r,s,v)`   | 控制者签名后，按合约规则构造 `_signature`         |
| 签名验证逻辑  | `ecrecover` 还原地址，比对一致即可 | 合约内部执行自定义验证逻辑                       |
| 验证接口    | 无需链上合约                  | `isValidSignature(hash, signature)` |
| 验证成功的标志 | 地址匹配                    | 返回 `0x1626ba7e`                     |

所以，ERC-1271 的本质是**把签名验证的逻辑从链下的密码学计算，转移到了链上的合约代码**，从而让智能合约也能像 EOA 一样“认可”一份消息。
