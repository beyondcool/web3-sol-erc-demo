# ERC-8004 Trustless Agents 教学示例

## 这个目录里有什么？

```
contracts/erc8004/
├── erc8004.md                    ← 全局总览（你在看的这个）
├── SimpleIdentityRegistry.sol    ← 身份注册表：Agent 的链上身份
├── SimpleReputationRegistry.sol  ← 声誉注册表：Agent 的链上信誉
└── ERC8004Test.t.sol             ← 教学测试：展示完整流程
```

## ERC-8004 是什么？三句话讲清楚

| 问题 | 答案 |
|------|------|
| **它解决什么问题？** | 在去中心化世界里，**如何发现和信任一个 AI Agent**？ |
| **怎么做？** | 三个**链上注册表**职责分明：**身份表**给 Agent 一个唯一的链上 ID（像营业执照），**声誉表**让用过的人打分（像大众点评），**验证表**让第三方确认 Agent 输出是否靠谱（像质检报告）。三表合一，无需信任 Agent 的"自我介绍"。 |
| **和别的协议什么关系？** | MCP/A2A 解决 Agent 之间怎么**通信**，ERC-8004 解决 Agent 怎么**被发现和信任**——互补关系。 |

## 两个核心注册表（本教学示例覆盖）

本示例实现了前两个注册表（第三个"验证注册表"较复杂，仅作概念介绍）：

```
┌──────────────────────────────────────────────────────────────────┐
│                    ERC-8004 生态系统                               │
│                                                                   │
│  ┌──────────────────────┐          ┌──────────────────────────┐   │
│  │   Identity Registry  │          │  Reputation Registry     │   │
│  │   （身份注册表）        │          │  （声誉注册表）            │   │
│  │                      │          │                          │   │
│  │  每个 Agent = NFT     │◀────────▶│  用户打分 → 链上声誉       │   │
│  │  TokenURI = 简介文件  │  查询    │  合约可读 → 可组合          │   │
│  │  NFT 所有者 = 控制者   │          │  可撤销 → 防恶意刷分        │   │
│  └──────────────────────┘          └──────────────────────────┘   │
│           ▲                               ▲                       │
│           │                               │                       │
│           └─────────── 注册 + 评分 ────────┘                       │
│                        （用户操作）                                 │
│                                                                   │
│  ┌──────────────────────────────────────────────────────────┐     │
│  │  ③ Validation Registry（验证注册表 — 未实现，仅概念）       │     │
│  │  让验证者（staker、zkML 证明者、TEE 预言机）确认 Agent 输出   │     │
│  └──────────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────┘
```

### Identity Registry（身份注册表）

**核心思想**：每个 AI Agent 都是一个 ERC-721 NFT。

| 概念 | 类比 | 在合约中 |
|------|------|----------|
| agentId | 营业执照号 | ERC-721 的 tokenId |
| agentURI | 公司简介页 URL | tokenURI，指向 JSON 注册文件 |
| owner | 公司法人 | NFT 持有者（可转让） |
| register() | 注册公司 | mint 一个新的 NFT |

**注册文件（Registration File）长什么样？**

agentURI 指向一个 JSON 文件，描述 Agent 的一切：

```json
{
  "name": "翻译助手",
  "description": "一个支持 50 种语言的 AI 翻译 Agent",
  "image": "https://example.com/avatar.png",
  "services": [
    { "name": "MCP", "endpoint": "https://translate-agent.example.com/mcp", "version": "2025-06-18" },
    { "name": "email", "endpoint": "translate@example.com" }
  ]
}
```

这个文件可以存在 IPFS、Web 服务器，或用 `data:` URI 编码后直接存链上。

### Reputation Registry（声誉注册表）

**核心思想**：用过 Agent 的用户可以给它打分，分数公开透明。

| 函数 | 谁调用 | 做什么 |
|------|--------|--------|
| `giveFeedback()` | 任何用户（除 Agent 所有者） | 给 Agent 打分（0-100）+ 标签 |
| `revokeFeedback()` | 之前打分的用户 | 撤销自己的评价 |
| `getSummary()` | 任何合约/用户 | 查询 Agent 的平均分 |

**防女巫（Sybil）说明**：合约层面不限制刷分，因为链上无法区分"100 个真实用户"和"1 个攻击者刷 100 次"。真正的声誉系统会：① 在链下给不同评价者加权；② 或要求评价者持有某种凭证（POAP、质押代币等）；③ 或过滤只信任特定白名单的评价者。

## 本教学示例的流程

```
Alice（开发者）                  IdentityRegistry           ReputationRegistry
    │                                │                           │
    │  ① 注册 Agent                  │                           │
    │  register("ipfs://...")        │                           │
    │──────────────────────────────▶ │                           │
    │                                │  铸造 NFT，分配 agentId    │
    │◀── Registered(agentId=1) ──────│                           │
    │                                │                           │
    │  ⚡ 现在 Alice 的 Agent 已       │                           │
    │     上线，等待用户使用和评价       │                           │
    │                                │                           │
    │  ② Bob 发现 Agent              │                           │
    │  getAllAgentIds() → [1]        │                           │
    │──────────────────────────────▶ │                           │
    │◀── agentId=1 ──────────────────│                           │
    │                                │                           │
    │  ③ Bob 查 Agent 详情           │                           │
    │  tokenURI(1)                   │                           │
    │──────────────────────────────▶ │                           │
    │◀── "翻译助手，MCP端点: ..." ─────│                           │
    │                                │                           │
    │  ═══ Bob 使用 Agent 服务 ═══    │                           │
    │  (链下：Bob 调用 Agent 的 API   │                           │
    │   完成了中英翻译任务)            │                           │
    │                               │                           │
    │  ④ Bob 根据使用体验打分         │                           │
    │  giveFeedback(1, 85, ...)     │                           │
    │──────────────────────────────────────────────────────────▶│
    │                               │                           │
    │  ⑤ Charlie 查声誉决定是否使用   │                           │
    │  getSummary(1)                │                           │
    │──────────────────────────────────────────────────────────▶│
    │◀── (count=1, avg=85) ─────────────────────────────────────│
    │                               │                           │
    │  Charlie："85 分，不错，用！"   │                            │
```

## 本教学示例 vs 真实 ERC-8004

| 方面 | 教学版 | 真实 ERC-8004 |
|------|--------|---------------|
| **agentWallet** | 未实现（直接跳过，降低复杂度） | 支持 EIP-712 签名验证设置钱包 |
| **Metadata** | 未实现 `getMetadata`/`setMetadata` | 预留链上元数据扩展 |
| **Feedback URI** | 未实现（评分+标签存链上即可演示核心概念） | 支持附带 URI 指向详细评价文件 |
| **Validation** | 仅概念介绍，未实现 | 完整的验证者请求/响应机制 |
| **聚合** | 简单算术平均 | 支持按评价者地址+标签过滤 |

## 推荐阅读顺序

1. **`SimpleIdentityRegistry.sol`** → Agent 身份注册（核心入口）
2. **`SimpleReputationRegistry.sol`** → 链上声誉系统
3. **`ERC8004Test.t.sol`** → 看测试 = 看完整交互流程
