## 传统交易 vs ERC-4337 的全流程对比

理解 ERC-4337 最好的方式：**把它看作一笔交易的"代理模式"**。

### 传统方式（EOA → 合约）

```
Alice (EOA)               Uniswap 合约
    │                         │
    │──── 交易(tx) ──────────▶ │
    │    from: Alice          │  执行 swap
    │    to: Uniswap          │
    │    data: swap(...)      │
    │    gas: 自付             │
    │    sig: Alice 签        │
```

**特点**：Alice 亲自发起交易、亲自付 gas、亲自签名。

### ERC-4337 方式（EOA → 智能钱包 → 目标合约）

```
Alice (EOA)                   EntryPoint              SimpleAccount         Uniswap
    │                             │                       │                    │
    │ ① 构造 UserOp               │                       │                    │
    │    sender: SimpleAccount     │                      │                    │
    │    callData: execute(Uniswap)│                      │                    │
    │    sig: Alice 签             │                      │                    │
    │                             │                       │                    │
    │──── UserOp ────────────────▶│                       │                    │
    │                             │                       │                    │
    │                             │ ② 验证阶段             │                    │
    │                             │──── validateUserOp ──▶│                    │
    │                             │    "Alice 签的吗？"    │ 检查签名=owner       │
    │                             │◀── ok ───────────────│                     │
    │                             │                       │                    │
    │                             │ ③ 执行阶段             │                    │
    │                             │──── execute(Uniswap) ▶│                    │
    │                             │                       │──── swap() ──────▶ │
    │                             │                       │                    │
    │                             │ ④ gas 结算            │                    │
    │                             │    从存款扣 gas        │                    │
```

**关键区别**：Alice 没有直接跟 Uniswap 交互。她**告诉 EntryPoint** 她想做什么，EntryPoint 协调她的智能钱包去执行。

## 四个核心角色

### 1. User（用户）—— 你

你有一个 EOA（存私钥的地方）和一个智能钱包（SimpleAccount 合约）。
你用 EOA 签名 UserOp，授权智能钱包去做事。

### 2. EntryPoint（协调器）—— 固定地址的中央交换机

它做三件事，按顺序：
1. **验证** — 问智能钱包"这个签名对吗？"
2. **执行** — 让智能钱包执行 UserOp 里的操作
3. **扣费** — 从钱包或 Paymaster 的存款里扣 gas

教学版的 EntryPoint 一次处理一个 UserOp（`handleOp`），真实版本是一次处理一批（`handleOps`）。

### 3. Account（智能钱包）—— 你的合约钱包

这是 ERC-4337 的核心创新：**你的钱包是一段代码，不是一对密钥**。

因为是代码，所以可以自定义一切：

| 能力 | 传统 EOA | 智能钱包 |
|---|---|---|
| 验证方式 | 只能用 ECDSA | **任意方式**（多签、Passkey、社交恢复） |
| 批量操作 | 一笔交易做一件事 | **一个 UserOp 可以触发多个操作** |
| Gas 支付 | 必须用 ETH | **可以用 USDC，或让别人代付** |
| 升级 | 不能 | **可以升级钱包逻辑** |
| 恢复 | 丢私钥=丢钱 | **可以设计恢复机制** |

教学版 SimpleAccount 只演示了最基本的 ECDSA 单签，但注释里解释了如何改成多签、社交恢复等。

### 4. Paymaster（代付者）—— 可选，但很强

Paymaster 解决一个实际问题：**新用户没有 ETH 怎么办？**

- 项目方部署一个 Paymaster
- 用户指定使用这个 Paymaster 付 gas
- Paymaster 验证通过后，gas 从 Paymaster 的存款扣除
- 用户不需要持有任何 ETH

教学版 Paymaster 是无条件代付。实际生产中可以加条件：
- 只有指定 DApp 的操作才代付
- 用户用 USDC "支付" gas（Paymaster 内部做 swap）
- 每个用户每天限额 N 笔

### 5. Bundler（打包者）—— 链下角色，本示例未实现

Bundler 是**链下**的独立角色，负责：
1. 收集用户提交的 UserOp
2. **在链下模拟执行**（Simulation）确认不会失败
3. 把多个 UserOp 打包成一笔交易提交给 EntryPoint
4. 赚取 gas 费

教学示例中，test 合约扮演了 Bundler 的角色——直接调用 `handleOp`。

## 教学版 vs 真实 ERC-4337

了解教学版简化了什么，有助于真正理解协议：

| 方面 | 教学版 | 真实 ERC-4337 |
|---|---|---|
| **处理方式** | `handleOp` 一次一个 | `handleOps` 批量处理 + `handleAggregatedOps` 签名聚合 |
| **Gas 计算** | 固定值 `100000 × 10gwei` | 动态计算：实际 gas 消耗 × EIP-1559 有效价格 |
| **存款管理** | 简单加减 | 复杂的存款、退款、stake、罚没机制 |
| **Simulation** | 没有（直接链上执行） | 链下严格模拟，防止 DoS |
| **Account 部署** | 假设已部署 | 支持 `initCode` 在第一个 UserOp 时部署钱包 |
| **Nonce 管理** | 简单递增 | 支持顺序 nonce 和任意 nonce 两种模式 |
| **安全性** | 基本签名验证 | 大量边界检查、重入防护、签名聚合验证 |

## 测试场景

| 测试 | 说明 |
|---|---|
| `test_BasicUserOpTransfer` | 完整流程：构造 UserOp → 签名 → 提交 → 转账成功 |
| `test_PaymasterSponsoredTransfer` | 钱包没存款，用 Paymaster 代付 gas |
| `test_RejectInvalidSignature` | 非 owner 签名被拒绝 |
| `test_ReplayProtection` | 同一笔 UserOp 不能执行两次（nonce 机制） |
